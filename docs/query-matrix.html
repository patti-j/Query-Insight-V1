Design Principles

1. Default 3-4 Tier 1 Tables

Each query should use 3-4 curated (DASHt_*) tables by default. This prevents "two-table SQL that lies" and ensures comprehensive answers.

2. Tier 1 vs Tier 2 Tables

Tier 1 (DASHt_*): Curated tables from Power BI dashboards. Pre-joined, optimized for common queries. Always preferred.

Tier 2 (Source): Raw source tables. Used only when Tier 1 lacks required detail or for complex JOINs.

3. Keyword Synonyms

Include common synonyms and abbreviations users might type:

  • Jobs: W/O, WO, work order, MO, mfg order
  • Operations: op, step, routing step
  • Late: overdue, past due, behind schedule, at risk
  • WIP: work in progress, in process, currently running

4. Override Rules

Some keywords trigger mandatory table inclusion. For example, "predecessor" questions always require DASHt_PredecessorOPIds + DASHt_Planning, overriding the normal matrix selection.

5. Context Hints

Some matrix entries include context hints that guide the LLM on how to structure JOINs and filters for complex query patterns (like "where used" queries).
